# v3.0.12 修复报告：金额提取将机器编号误当金额

## 问题

用户上传一张 **12元 广东增值税电子普通发票**，系统识别金额为 **499,098,814,701.00**（约4990亿元）。

实际金额 `¥12.00` 被机器编号 `499098814701`（12位纯整数）完全替代。

## 根因

**数电票 PDF 文本提取顺序与视觉布局不一致。**

```
PDF文本流中的位置：
┌─ 位置91:   "价税合计 (大写）"
│ ... (500+字符的其他字段) ...
├─ 位置598:  "（小写）¥ 12.00"     ← 正确答案
│
├─ 某处:     "机器编号：499098814701"  ← 被误当金额！
└─

失败链路：
策略0(小写¥, 窗口300) → 12.00在598,超出范围 → 失败
策略1(价税合计max¥, 窗口200) → 同上 → 失败
策略2(clean匹配, 窗口100) → 同上 → 失败
策略3(合计行正则) → 匹配到"合              计"后紧跟的数字 → 命中机器编号 → 返回错误值 ❌
```

## 修复（6项改动）

| # | 改动 | 说明 |
|---|------|------|
| 1 | **窗口扩大** | 策略0/1/2 窗口从 300/200/100 → 2000/2000/200 |
| 2 | **新增策略1.5** | 从"价税合计"搜索到文档末尾找 `(小写)¥` |
| 3 | **机器编号排除** | 预处理识别"机器编号"区域数字，所有策略候选均过滤 |
| 4 | **策略3加固** | 要求 `.XX` 格式 + 长度≤10位，否则丢弃 |
| 5 | **策略4加固** | 要求小数点 + 长度≤12位 + 排除机器编号 |
| 6 | **AI Prompt加固** | 明确禁止把机器编号当金额，列出5种常见误判陷阱 |
